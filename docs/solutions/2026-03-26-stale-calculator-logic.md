# 2026-03-26 Решение рассинхронизации UI компонентов с D-объектом

## Управленческое резюме
При добавлении 6-й страховой компании (Generali) и разбиении тарифов на под-опции (Wiener M1/M2/M3), таблица сравнения (`updateTable()`) обновилась корректно (т.к. она динамически обходит `D`), однако Калькулятор рекомендаций (`calculate()`) и блок Сценариев (`SCENARIOS`) продолжали выдавать устаревшую и неполную информацию.

## 🔴 Problem
Калькулятор (JS-функция `calculate()`) давал некорректные и скрыто-устаревшие рекомендации пользователям:
1. Полностью игнорировал новую компанию Generali.
2. Использовал старую цену Dunav (~€65-100/год), хотя в главном объекте она уже была повышена до ~€110/год.
3. Ошибочно рекламировал "стоматологию" у пакета UNIQA, хотя в новом `D` объекте это было разжаловано до "⚠️ Только хитна (экстренная)".

## 🔍 Root Cause
Архитектурный изъян (hardcoding). В то время как отрисовка таблицы динамически обходит источник истины — JS-объект `D`, алгоритм калькулятора был реализован через жёстко закодированные `recs.push({name: '...', price: '...', note: '...'})`. Изменение структуры `D` (ключей, цен) никак не валидировалось и не линковалось с `calculate()`. Технический долг привел к тому, что сайт на одной странице показывал две разные цены для одного пакета.

## 🛠️ Solution
Был проведён аудит скрипта (метод `/critic`).
Функция `calculate()` была полностью переписана с учётом ВСЕХ 6 компаний и новых sub-опций. 
Временно оставлен хардкод рекомендаций, чтобы сохранить "человечные" описания ("Только для ВНЖ", "Единственная компания с разовой выплатой"), но данные 100% выверены с D-объектом.

## 🛡️ Prevention
Для предотвращения подобного в будущем (quality gate для архитектуры):
1. **При любом изменении цен в `D-объекте`**, необходимо выполнять полнотекстовый поиск (Ctrl+F) по старой цене во всём `index.html`, чтобы найти захардкоженные вхождения в `calculate()`, `MARKO_TIPS` и `html`-разметке (footer).
2. **В идеале (на будущее):** Переписать `calculate()` так, чтобы функция собирала price и название директивно из `D.company.package.price`, а в хардкоде оставлять только строку `note` и `score`. Это предотвратит дрейф цен.
